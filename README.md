# Сбор и обработка данных Discord

## Описание проекта

Этот проект — попытка собрать в одном месте данные о поведении пользователей Discord (в первую очередь — голосовая активность и онлайн) и информацию о серверах, чтобы потом использовать их для анализа: когда люди активны, как формируются группы, какой “праймтайм” у серверов и т.д.

Данные приходят через:
1. Discord API и discord.py (раз в 30 минут)
2. Ежедневный парсинг через модифицированный [OSINTCord](https://github.com/MrBoombastic/OSINTCord) 

Немного обрабатываются и складываются в PostgreSQL, откуда делаются выборки для Excel и для визуализации в Metabase.

## Как это работает в целом

У меня есть два отдельных скрипта, которые запускаются по расписанию:

### 1. `discord_bot.py` (каждые 30 минут)

Этот скрипт работает через Discord API.

Что он делает:

* проходит по списку серверов
* получает список голосовых каналов
* смотрит, кто сейчас находится в голосе
* дополнительно собирает:

  * активность пользователей (во что играют)
  * общее количество участников сервера

Данные собираются в память и потом пачкой отправляются в PostgreSQL.

Важно:

* используются асинхронные запросы (`aiohttp`)
* есть обработка rate limit (ответ 429)
* есть retry при сетевых ошибках

По сути это источник “почти онлайн” данных.

---

### 2. `discord_exporter.py` (раз в день)

Этот скрипт работает с логами, которые генерирует отдельный Node.js инструмент (OSINTCord).

Пайплайн такой:

* сначала запускается внешний процесс (`npm start`)
* он генерирует `.txt` файлы с данными
* дальше Python-скрипт читает эти файлы и обрабатывает их

Что происходит при обработке:

* читаются все файлы из папки `/logs`
* данные объединяются в одну таблицу
* добавляются служебные поля (время, server_id)
* чистятся роли (убираются лишние префиксы)
* удаляются строки с ботами
* приводятся типы и значения (`NaN → None`)

После этого данные загружаются в PostgreSQL.

Отдельно:

* исходные файлы после обработки удаляются (чтобы не обрабатывать повторно)

---

## Куда всё сохраняется

Все данные складываются в PostgreSQL (одна база, несколько таблиц):

* `voice_users` — кто и в каком голосовом канале был
* `user_activity` — во что играют пользователи
* `server_info` — размер серверов
* `user_guild` — данные из логов (роли, участие и т.д.)

Вставка делается через batch (`executemany`), чтобы не делать по одному insert.

---

## Что происходит дальше (аналитика)

После загрузки данных:

* выполняется SQL-запрос (лежит в отдельном `.sql` файле)
* результат забирается в pandas
* и отправляется в Google Sheets

Это нужно, чтобы быстро смотреть данные и строить графики без поднятия отдельного BI.

---

## Поток данных (по шагам)

Если упростить, то выглядит так:

1. Discord API → `discord_bot.py` → PostgreSQL
2. OSINTCord → `.txt` файлы → `discord_exporter.py` → PostgreSQL
3. PostgreSQL → SQL → pandas → Google Sheets

## Планирование

Оба скрипта запускаются автоматически:

* `discord_bot.py` — каждые 30 минут
* `discord_exporter.py` — раз в день

Используется системный планировщик задач.

---

## Почему сделано именно так

Разделил на два источника, потому что:

* через API удобно получать текущее состояние (кто сейчас в голосе)
* через логи можно получить более “тяжёлые” и подробные данные

В итоге получается более полная картина.

## Зачем это вообще

Основная цель — понять поведение пользователей:

* когда сервер “живёт”
* сколько людей одновременно в голосе
* как формируются группы
* какие есть паттерны активности

Это можно использовать, например, для:

* планирования активности
* роста сообщества
* координации внутри игровых команд
